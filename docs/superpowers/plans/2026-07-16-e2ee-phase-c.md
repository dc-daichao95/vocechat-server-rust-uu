# E2EE Phase C — Server API (2026-07-16)

## Delivered

| Item | Detail |
|------|--------|
| `GET /user/e2e/protocol` | Returns `e2e_available`, `e2e_default_on`, `e2e_protocol_ver` |
| `PUT /user/e2e/identity` | When `e2e_protocol_ver >= 2`, requires non-empty `signed_prekey_pub` + `signed_prekey_sig` |
| `GET /user/e2e/bundle/:uid` | When protocol ≥ 2, rejects identities missing signed prekey (`E2E_SIGNED_PREKEY_REQUIRED`) |
| Send path | When protocol ≥ 2, rejects `e2e_ver < required` → `E2E_UPGRADE_REQUIRED` |
| Device link | Migration + start / package / complete / status |
| Backup | Unchanged (`PUT/GET /user/e2e/backup`) |

## Device-link flow

1. Logged-in device: `POST /user/e2e/device-link/start` → `{ link_id, token, expires_at }` (~10 min)
2. Same device: `PUT /user/e2e/device-link/:link_id/package` with opaque `package_base64`
3. New device: `POST /user/e2e/device-link/complete` `{ token }` → `{ uid, package_base64 }` (once)
4. Owner poll: `GET /user/e2e/device-link/:link_id` → `pending|ready|consumed|expired`

Server stores **only opaque bytes**; never private keys.

## Cutover

Admin sets login config `e2e_protocol_ver = 2` when Web + Flutter ship v2 send. Until then default remains `1` so existing clients keep working.
